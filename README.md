# Auth0 + ORDS + ADB Companion

Reference repository for a technical blog post on securing Oracle REST Data Services (ORDS) endpoints in Oracle Autonomous Database (ADB) with Auth0-issued JWT access tokens.

This repository is for Oracle architects, Oracle developers, ADB and ORDS practitioners, and technical engineers who understand OAuth2 at a basic level but want a concrete example of how Auth0, ORDS, and ADB fit together.

## What This Repo Covers

- Auth0 as the authorization server and JWT issuer
- ORDS as the resource server
- ADB as the host for the REST-enabled module and data logic
- Example ORDS JWT trust settings
- Example SQL for a minimal secured endpoint
- Example shell scripts for token retrieval, endpoint invocation, and local JWT inspection
- Troubleshooting guidance for common integration failures

## What This Repo Does Not Cover

- Provisioning a real Auth0 tenant
- Provisioning a real Autonomous Database instance
- Production-grade secret management
- Frontend implementation
- A generic ORDS framework

## Architecture Summary

The trust model is intentionally simple:

1. A client requests a machine-to-machine access token from Auth0.
2. Auth0 issues an RS256-signed JWT access token with an `iss` claim, an `aud` claim, and one or more scopes.
3. The client calls an ORDS endpoint with that access token.
4. ORDS validates the token against the configured issuer, audience, and JWKS URL.
5. ORDS applies its own privilege model before allowing access to the module handler.
6. ADB returns the response generated by the REST-enabled SQL or PL/SQL handler.

The most important configuration rule is that the Auth0 API Identifier becomes the JWT `aud` claim. It is a logical audience string, not the ORDS endpoint URL, and it must exactly match the audience configured in ORDS.

## Prerequisites

- An Auth0 tenant where you can create an API and a machine-to-machine application
- An ORDS deployment configured to access an Autonomous Database schema
- A schema where you can run the demo SQL scripts
- `curl`
- `jq`
- A POSIX shell or portable Bash environment

## Quick Start

1. Follow [docs/auth0-setup.md](docs/auth0-setup.md) to create an Auth0 API, choose an API Identifier, create a machine-to-machine application, and grant scopes.
2. Review [docs/ords-jwt-profile.md](docs/ords-jwt-profile.md) and adapt [ords/jwt-profile.example.properties](ords/jwt-profile.example.properties) to your ORDS environment.
3. Copy `.env.example` to `.env` locally and populate the required values. If your schema is already ORDS-enabled, set `ORDS_SCHEMA` to the existing schema base path instead of assuming `demo`.
4. Run the SQL scripts in order:
   - `sql/01_create_sample_objects.sql`
   - `sql/02_define_module.sql`
   - `sql/03_define_security.sql`
5. Request an access token with `scripts/get-token.sh`.
6. Call the secured endpoint with `scripts/call-secure-endpoint.sh`.
7. If the call fails, use `scripts/decode-jwt.sh` and [docs/troubleshooting.md](docs/troubleshooting.md).

## Test Flow

The intended path through this repository is:

1. Configure Auth0.
2. Configure ORDS.
3. Populate local `.env`.
4. Apply SQL.
5. Request a token.
6. Call the secured endpoint.
7. Verify the result.

Expected high-level outcomes:

- `200` when the token is valid and ORDS authorization allows the call
- `401` when the token is missing, malformed, expired, signed incorrectly, or does not match issuer or audience expectations
- `403` when the token is valid but the request does not satisfy the ORDS privilege model

## Security Notes

- Do not commit `.env`, client secrets, wallet files, or any other local credentials.
- The scripts in this repository are demo helpers, not hardened production integrations.
- The decode helper does not verify signatures; it only decodes the JWT header and payload locally.
- Production deployments usually need stronger secret handling, vault integration, network controls, API gateways, and more advanced authorization logic.
